# PRD: 官网 UI 重构

## 0. 文档目的

本文档用于指导中科数测官网的 UI 视觉重构工作，面向产品负责人、设计师、前端开发、测试和后续 BMAD 工作流。本文档基于 `docs/source-site/urls.md`、`docs/source-site/page-inventory.md`、`docs/rebuild/prd-prep.md` 生成，约束重构团队在不新增页面、不新增内容、不改变页面逻辑的前提下，只调整 UI 视觉、版式、组件样式、动效和响应式表现。

本文档是后续 `bmad-ux`、`bmad-architecture`、`bmad-create-epics-and-stories` 的输入，不替代视觉设计稿、技术架构或开发 Story。所有页面、文案、链接、SEO 和素材保留要求以源站盘点为准。

## 1. 背景与目标

现有官网已上线，覆盖首页、产品详情页、解决方案页、技术服务页和公司介绍页。当前任务不是重新定义官网内容，也不是新增产品或业务，而是对现有官网进行视觉和前端呈现层重构，使页面在专业度、信息层级、响应式体验和组件一致性上达到更高质量。

本次重构的核心目标是：保留旧官网的页面范围、内容结构、文案含义、导航逻辑和 CTA 路径；重新设计页面视觉风格、排版节奏、组件系统、移动端适配和交互反馈；最终形成可开发、可验收、可持续迭代的新官网前端基础。

## 2. 目标用户与使用场景

### 2.1 目标用户

- 企业客户决策者：快速理解公司能力、产品矩阵、解决方案和可信度。
- 安全技术负责人：评估 Wisdom、Swift AI、Swift Code、Swift SCA、Wisdom Lens、Swift Pentest 等产品能力。
- 行业解决方案客户：查看网联汽车、特种装备等场景方案，判断是否适配自身建设需求。
- 潜在线索用户：通过“免费试用”“咨询方案”“联系我们”等 CTA 进入咨询路径。
- 内部运营和销售团队：复用官网页面作为对外展示和客户沟通入口。

### 2.2 Jobs To Be Done

- 当访客第一次进入官网时，需要快速判断公司定位、核心安全能力、产品矩阵和解决方案适配度。
- 当访客查看某个产品详情页时，需要清楚理解产品用途、核心功能、优势、集成能力和咨询入口。
- 当访客查看解决方案页面时，需要理解场景问题、方案组成、价值优势和适用业务场景。
- 当访客准备进一步沟通时，需要稳定找到联系、试用或咨询入口。
- 当内部团队维护官网时，需要新页面结构足够统一，避免每个页面单独堆样式。

### 2.3 关键用户旅程

- **UJ-1. 企业客户从首页判断是否继续了解产品。** 访客打开首页，先看到公司定位和安全能力，再浏览核心产品、行业解决方案、关于公司和成功案例，最终点击“免费试用”或“了解更多”。价值达成点是访客能在一个页面内建立对公司业务范围和能力矩阵的基本信任。
- **UJ-2. 安全技术负责人查看产品详情。** 访客从导航或首页产品入口进入某个产品详情页，依次阅读首屏定位、产品功能、产品优势、集成能力和 CTA。价值达成点是访客能判断该产品是否符合协议安全、模型安全、代码安全、供应链安全或渗透测试需求。
- **UJ-3. 行业客户查看解决方案并发起咨询。** 访客进入网联汽车或特种装备解决方案页，浏览方案概述、方案组成、价值优势、业务场景，最终点击“咨询方案”。价值达成点是访客确认该方案可以支撑其合规、实验室建设或安全验证需求。
- **UJ-4. 访客查看公司可信度。** 访客进入公司介绍页，浏览公司介绍、研发实力、企业文化、荣誉资质、分布地区、FAQ 和 CTA。价值达成点是访客能建立基本组织信任，并找到进一步联系入口。

## 3. 术语表

- **源站** — 当前已上线的 `https://www.zksctest.com/` 官网页面。
- **页面盘点** — `docs/source-site/page-inventory.md` 中记录的页面范围、模块、链接、图片、SEO、表单识别和不可改动项。
- **UI 重构** — 在不改变业务内容和页面逻辑的前提下，调整视觉、版式、组件、动效和响应式表现。
- **页面逻辑** — 导航结构、按钮跳转、CTA 路径、表单入口、锚点切换、页面模块关系和用户路径。
- **CTA** — 引导访客行动的按钮或链接，包括“免费试用”“联系我们”“咨询方案”“了解更多”。

- **解决方案页** — 网联汽车解决方案页面和特种装备解决方案页面。
- **不可改动项** — 页面定位、内容范围、原有按钮文字和跳转目标、产品/方案/服务信息结构。

## 5. 功能需求

### 5.1 页面范围与内容保留

**FR-1: 页面清单锁定**  
重构项目必须覆盖 §4.1 列出的 11 个页面。任何新增页面、删除页面或合并页面都必须先经过用户确认。

**Consequences:**
- 验收时逐页检查 11 个页面是否都有新版本。
- 页面缺失、页面类型错位、页面内容被并入其他页面均视为不通过。

**FR-2: 文案含义保留**  
重构后的每个页面必须保留源站对应页面的核心文案含义、产品定位、方案描述、服务描述和 CTA 文案。

**Consequences:**
- 不得擅自新增产品卖点、行业、客户案例、资质或数据。
- 可为排版做轻微断句、分组、标题层级调整，但不得改变原意。
- 如果需要修正文案、品牌名或公司定位，必须作为独立内容变更审批，不混入 UI 重构。

**FR-3: 页面模块保留**  
每个页面必须保留源站已盘点的主要模块。例如首页必须保留首屏、合作生态、核心安全能力、核心产品介绍、行业解决方案、关于公司、成功案例和底部 CTA；产品详情页必须保留首屏、产品功能/价值、产品优势、集成/CTA 等源站模块。

**Consequences:**
- 模块顺序默认不改变。
- 如为了响应式体验调整模块内部布局，必须不改变模块含义和用户路径。

### 5.2 导航、链接与 CTA

**FR-4: 导航结构保留**  
新官网必须保留源站导航中的产品、解决方案、服务、公司介绍和联系入口。导航文字和目标链接不得擅自改写。

**Consequences:**
- 产品导航必须保留 Wisdom、Swift AI、Swift SCA、Swift Code、Wisdom Lens、Swift Pentest 等入口。
- 解决方案导航必须保留网联汽车和特种装备入口。
- 服务导航必须保留三方测评、渗透测试等现有入口。

**FR-5: CTA 路径保留**  
所有“免费试用”“联系我们”“咨询方案”“了解更多”等 CTA 必须保持源站业务含义和目标路径。

**Consequences:**
- 指向 `contact.html` 的 CTA 不得改为其他路径。
- 指向解决方案页、公司介绍页、产品页的“了解更多”不得改为无效锚点。
- 如果新视觉中改变按钮位置，必须保证桌面端和移动端都可见、可点击。

**FR-6: 页脚与外部链接保留**  
页脚中的备案、外部社区/媒体链接、站内导航链接必须保留，不得因视觉简化而删除。

### 5.3 页面级 UI 重构要求

**FR-7: 首页 UI 重构**  
首页需要重新组织视觉层级，使公司定位、核心安全能力、产品矩阵、解决方案、公司介绍、成功案例和底部 CTA 在首屏到后续模块中形成清晰阅读节奏。

**Consequences:**
- 首屏必须保留“致力于成为全球领先的低空与商业航天网络安全方案提供商”相关定位。
- 核心安全能力和核心产品介绍必须可快速扫描。
- 行业解决方案的“网联汽车”“特种领域/特种装备”入口必须明确。
- 成功案例和合作生态不得被删除。

**FR-8: 产品详情页模板重构**  
六个产品详情页必须采用统一但可区分的详情页结构，保留每个产品的首屏定位、功能、优势、集成能力和底部 CTA。

**Consequences:**
- Wisdom 页面保留模糊测试、协议安全、漏洞发现和专有协议测试表达。
- Swift AI 页面保留模型安全、多模态检测、LLM 安全评测与防护、合规评估表达。
- Swift Code 页面保留源代码审计、双引擎检测、多语言支持、产品优势表达。
- Wisdom Lens 页面保留灰盒模糊测试、覆盖率反馈、测试驱动生成、缺陷挖掘表达。
- Swift SCA 页面保留 SBOM、漏洞管理、合规、供应链安全表达。
- Swift Pentest 页面保留攻击面识别、漏洞扫描与验证、智能化渗透测试、风险报告表达。

**FR-9: 解决方案页 UI 重构**  
网联汽车和特种装备解决方案页必须突出方案概述、方案组成、价值优势、业务场景和咨询 CTA。

**Consequences:**
- 网联汽车页面必须保留 GB 44495-2024、UNECE R155、实验室工具链、TARA、固件/协议/整车/云端验证等源站信息。
- 特种装备页面必须保留脆弱性、渗透性、生存性、自主可控实验能力、工具链和整改闭环等源站信息。
- 业务场景中的 tab/锚点逻辑如继续使用，必须保持可访问、可点击、移动端可用。

**FR-10: 技术服务页 UI 重构**  
渗透测试服务页必须保留服务定位、测试说明、服务优势、检测内容和联系 CTA。

**FR-11: 公司介绍页 UI 重构**  
公司介绍页必须保留公司定位、关于公司、研发实力、企业文化、荣誉资质、公司分布地区、FAQ 和底部 CTA。

### 5.4 视觉系统与响应式

**FR-12: 建立统一 UI 组件规则**  
重构必须形成统一的导航、按钮、卡片、标签、产品能力块、方案模块、数据指标、页脚和 CTA 组件样式。

**Consequences:**
- 同类按钮样式统一，主按钮、次按钮、文本链接层级清楚。
- 卡片不能无节制堆叠，信息密度要符合企业官网阅读习惯。
- 页面不使用只有装饰意义、无法承载产品信息的视觉元素。

**FR-13: 桌面端响应式要求**  
桌面端布局需要适配常见宽屏和笔记本视口，保证导航、首屏、产品卡片、方案内容和 CTA 不重叠、不溢出。

**FR-14: 移动端响应式要求**  
移动端必须支持导航折叠、模块纵向阅读、按钮可点击、图片不遮挡文字、长标题自动换行、内容不横向溢出。

**FR-15: 交互状态保留和优化**  
导航展开、hover、按钮状态、tab 切换、FAQ 展开、CTA 点击等交互状态必须保持业务逻辑不变，并提升视觉反馈清晰度。

### 5.5 SEO、素材与上线约束

**FR-16: SEO 信息保留**  
新页面必须保留每页源站 SEO title、description、keywords；源站未识别到的页面需要在验收前确认是否补齐。

**FR-17: 图片素材迁移和替换规则**  
源站图片、logo、产品截图、图标、客户/院校/行业 logo 等素材必须建立素材清单。默认保留源站素材；如替换，必须确认不改变内容含义。

**FR-18: URL 结构保留**  
新站默认保持 `.html` URL 结构和站内链接目标不变。[ASSUMPTION: 上线时需要兼容旧 URL，以避免 SEO 和外部链接损失。]

**FR-19: HTTPS 证书问题单独处理**  
源站 HTTPS 证书当前已过期。该问题不属于 UI 重构功能需求，但上线前必须作为运维风险单独修复。

## 6. 非功能需求

- **可访问性**：文本对比度、按钮可点击区域、焦点状态、图片 alt、表单/FAQ/导航交互需要满足基础可访问性要求。
- **性能**：新页面应减少无意义大图、重复动画和阻塞脚本；首屏关键内容应优先加载。
- **兼容性**：支持主流 Chrome、Safari、Edge；移动端支持 iOS Safari 和 Android Chrome。
- **可维护性**：页面模板、组件样式、颜色、字体、间距需要集中管理，避免每页复制一套样式。
- **可验收性**：每个页面必须可通过旧站截图、页面盘点、链接清单和移动端检查进行对照验收。

## 7. 非目标

- 不重新定位公司品牌。
- 不新增产品、服务、行业解决方案或客户案例。
- 不新增官网后台。
- 不改写业务文案含义。
- 不改变导航、CTA、表单或链接的业务路径。
- 不把未纳入范围的 `contact.html`、`datacheck.html`、`softwareassess.html`、`service-secas.html` 等页面一起重构。
- 不处理 HTTPS 证书修复、服务器部署、域名解析等运维事项，除非后续单独立项。

## 8. MVP 范围

### 8.1 In Scope

- 11 个已确认页面的 UI 视觉重构。
- 统一导航、页脚、按钮、卡片、CTA 和页面模块样式。
- 桌面端和移动端响应式。
- 源站文案、链接、SEO、图片素材迁移和保留。
- 页面级验收标准与后续 Story 拆分输入。

### 8.2 Out of Scope for MVP

- 新增内容运营系统或 CMS。
- 新增后台管理。
- 新增多语言。
- 新增动画大片、视频、3D 场景或复杂前端交互。
- 对未纳入范围页面进行 UI 重构。
- 产品命名、公司定位、业务介绍的内容策略调整。

## 9. 页面级验收标准

### 9.1 通用验收

- 11 个页面均已完成新版本。
- 页面文案含义与源站一致。
- 页面主要模块与源站一致。
- 导航、页脚、CTA、外部链接可点击且目标正确。
- 页面在桌面端和移动端无明显文字重叠、按钮溢出、图片遮挡、横向滚动。
- SEO title、description、keywords 按源站保留；源站缺失项单独列为待确认。
- 所有源站 CTA 的业务路径保留。

### 9.2 页面类型验收

- 首页：首屏、合作生态、核心安全能力、核心产品、行业解决方案、关于公司、成功案例、底部 CTA 均存在。
- 产品详情页：六个产品均保留首屏定位、功能、优势、集成/CTA；模板统一但产品差异清楚。
- 解决方案页：网联汽车和特种装备均保留方案概述、组成、价值优势、业务场景和咨询 CTA。
- 技术服务页：渗透测试服务保留服务说明、优势、检测内容和联系入口。
- 公司介绍页：公司介绍、文化、荣誉、分布地区、FAQ 和联系入口均保留。

## 10. 成功指标

**Primary**

- **SM-1**：页面范围完整率 100%。验证 FR-1。
- **SM-2**：源站核心内容保留率 100%。验证 FR-2、FR-3。
- **SM-3**：CTA 和导航链接可用率 100%。验证 FR-4、FR-5、FR-6。
- **SM-4**：桌面端和移动端关键页面无阻塞级视觉问题。验证 FR-13、FR-14。

**Secondary**

- **SM-5**：同类页面模板复用度提升，六个产品详情页具备统一视觉结构。验证 FR-8、FR-12。
- **SM-6**：页面 SEO 信息保留完整。验证 FR-16。

**Counter-metrics**

- **SM-C1**：不要为了视觉冲击牺牲内容完整性。若页面更美观但删减源站关键模块，视为失败。
- **SM-C2**：不要为了统一模板抹平产品差异。六个产品详情页必须保留各自能力重点。

## 11. 风险与缓解

- **风险：视觉改版过程中改写了原文案含义。** 缓解：开发前先做页面级文案锁定，验收时逐页对照。
- **风险：只看桌面端，移动端出现遮挡和溢出。** 缓解：每个 Story 必须包含移动端验收。
- **风险：导航中存在未纳入范围页面，开发时误删链接。** 缓解：未纳入范围页面不重构，但链接必须保留。
- **风险：源站证书过期影响后续采集或上线。** 缓解：单独创建运维修复项，不混入 UI 重构需求。
- **风险：源站 HTML 未识别到 form 标签，但实际可能存在第三方表单或弹窗。** 缓解：补充截图和人工交互核验。
